# Documentation Reference Convention

## Overview

Docs generated by ADD commands reference each other via **stable IDs** (`F0042`, `H0013`, `PRD0009`, `CHG0001`) rather than paths or titles. References use the `{{doc:<ID>}}` syntax and are resolved at **runtime** by agents via grep on frontmatter `id:` — NOT at build time. This is the key contrast with `{{cmd:}}` / `{{skill:}}`, which are resolved by `build.js` during provider build.

Cross-link: `{{skill:add-resource-path-convention/SKILL.md}}` covers build-time command/skill refs. This skill covers runtime doc refs. They are complementary and both mandatory.

## When to Use

- Writing a generated doc (about.md, plan.md, design.md, changelog, etc.) that mentions another doc
- Populating `related: []` in frontmatter per `{{skill:add-doc-schemas/SKILL.md}}`
- Cross-linking in the body of a doc consumed by agents
- Allocating a new ID for a feature, hotfix, PRD, or changelog entry

## When NOT to Use

- External docs, vendor READMEs, URLs — use plain Markdown links
- Drafts, brainstorms outside the `brainstorm` schema, scratch notes
- Command/skill/script references — use `{{cmd:}}` / `{{skill:}}` / literal `.codeadd/scripts/*`
- Line-level code references — use `path:line` paths
- PRD cross-refs inside `/add.strategy` output where the PRD itself is the primary artefact (still ok to use IDs, but not required)

## ID Convention

Every ADD doc type that participates in the reference graph has a prefix. IDs are **global within their prefix** and zero-padded to 4 digits.

| Prefix | Type | Format | Example | Allocator |
|---|---|---|---|---|
| `F` | Feature | `F[NNNN]` | `F0042` | `/add.new` |
| `H` | Hotfix | `H[NNNN]` | `H0013` | `/add.hotfix` |
| `PRD` | PRD (framework strategy) | `PRD[NNNN]` | `PRD0009` | `/add.strategy` |
| `CHG` | Changelog entry | `CHG[NNNN]` | `CHG0001` | `/add.done` |

Other doc types (`STACK`, `OWNER`, `PRODUCT`, `AUDIT-<date>`, `DIAG-<slug>`, `COPY-<slug>`, `LAND-<slug>`, `BRN-<slug>`) use fixed or slug-based IDs instead of numeric counters — see `{{skill:add-doc-schemas/SKILL.md}}` for the per-schema ID rule.

## ID Allocation

IDs are allocated **manually** per prefix, one counter per prefix, monotonically increasing. Same model PRDs already use today.

- Before creating a new doc, grep the counter: `grep -rhE "^id: F[0-9]{4}$" docs/ | sort -u | tail -1`
- Use the next integer, zero-padded.
- Never reuse an ID even if the doc is deleted.
- Never reassign IDs across prefixes.

Wave 3 of PRD0009 will extend `.codeadd/scripts/status.sh` with a `next-id <PREFIX>` subcommand that automates this lookup. Until then, allocation is manual and the responsibility of the generator command.

## Reference Syntax

Two forms. Both are authoritative.

**Inline reference (body text):**

```markdown
See {{doc:F0042}} for the feature scope.
The regression was introduced by {{doc:H0013}}.
Aligned with {{doc:PRD0009}}.
```

**Frontmatter reference (`related:`):**

```yaml
---
id: F0043
type: feature-plan
related: [F0042, PRD0009, H0013]
---
```

`related:` is a YAML list of bare IDs (no `{{doc:}}` wrapping inside YAML). Body text MUST wrap IDs in `{{doc:...}}` so agents can distinguish refs from incidental text.

## Resolution Model

Resolution is **runtime**, not build-time. When an agent encounters `{{doc:F0042}}`:

1. Run `grep -rE "^id: F0042$" docs/`
2. Resolve to the file path containing that frontmatter line
3. Read the target doc (frontmatter + TL;DR first, sections on demand)

Contrast with build-time variables:

| Syntax | Resolution | Resolver |
|---|---|---|
| `{{cmd:NAME}}` | Build time | `framwork/build.js` |
| `{{skill:NAME/FILE}}` | Build time | `framwork/build.js` |
| `{{doc:ID}}` | **Runtime** | Agent via grep |

**`build.js` MUST NOT be extended to resolve `{{doc:}}`.** Doing so would couple the build pipeline to user doc content and defeat the JIT loading model. The string `{{doc:<ID>}}` MUST pass through build untouched.

## Orphan Handling

An orphan ref is a `{{doc:<ID>}}` whose target does not resolve (no matching `^id: <ID>$` in `docs/`).

- Orphan refs are **warnings**, not errors. A generator command does NOT fail the gate on an orphan.
- The validation gate in `{{skill:add-doc-schemas/SKILL.md}}` surfaces the list of unresolved refs at the end of command output.
- The user (or a follow-up command) decides whether to fix the ref, create the missing doc, or accept the orphan as intentional (e.g., forward reference to an unplanned hotfix).
- Deleting a doc does NOT retroactively fix inbound refs — they become orphans by design. Prefer marking the doc `status: deprecated` and keeping it addressable.

## Edit Discipline

Manual edits to generated docs are allowed but MUST preserve the contract:

- Always bump `updated:` in frontmatter on any edit.
- Optionally set `by:` to signal the editor (`human` or an agent name). `/add.audit` may flag docs whose `by:` drifts from the expected generator.
- Never change `id:` once allocated. If the doc fundamentally changes type, create a new doc with a new ID and set `related:` back to the old one.
- Never edit `related:` to hide a broken link — fix the link or accept the orphan warning.
