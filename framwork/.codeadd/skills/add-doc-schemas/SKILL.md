# Documentation Schemas Registry

## Overview

Canonical registry of schemas for every documentation asset generated by ADD commands. Commands MUST load the relevant schema via `{{skill:add-doc-schemas/SKILL.md}}` just-in-time instead of inlining templates. All generated content is **extractive only** — no abstractive summarization; preserves auditability and grammar per Jha et al.

Agents consuming docs read frontmatter + TL;DR first, then grep specific section anchors on demand. This skill is the single source of truth for doc structure across the framework.

## When to Use

- Writing or refactoring a command that creates/updates a doc asset consumed by agents
- Generating: about.md, plan.md, design.md, changelog, owner.md, product.md, audit/diagnose reports, saas-copy, landing-page specs, hotfix docs
- Validating an existing doc against its schema (via the gate block below)
- Creating new generator commands that produce agent-consumed artefacts

## When NOT to Use

- Free-form drafts or scratch notes
- `/add.brainstorm` exploratory outputs beyond the `brainstorm` schema (raw transcripts stay free)
- `/add.strategy` outputs — meta-process for evolving the framework itself, keeps its own template
- External docs, vendor READMEs, third-party specs
- PRDs authored by `/add.strategy` (own lifecycle)
- Commit messages, PR bodies (see `add-commit`)

## Core Principles

- **Extractive only** — no abstractive summarization in generated docs. Allowed forms: tables, bullet lists, minified JSON, topic sentences, factual bullets. Prohibited: narrative paraphrase of source material.
- **JIT loading** — every doc MUST start with frontmatter + TL;DR (~50 words). Agents load full sections only when needed via grep by anchor.
- **Stable IDs** — every doc MUST carry an `id:` in frontmatter. ID format and resolution delegated to `{{skill:add-doc-ref-convention/SKILL.md}}`.
- **Heterogeneous zone budgets** — compression varies by zone: Spec = zero compression (exact), Rationale = extractive ≤200 words, Examples = 1 canonical example, History = factual bullets only.
- **Section-chunks ~100-150 words** (~128 tokens) — each opens with a topic sentence to enable grep-as-retrieval.
- **TOC required if >3 sections** — flat bullet list after TL;DR, linking anchors.
- **No emojis in headers.** No decorative ASCII. Minified JSON for structured data.

## Universal Document Requirements

Every schema inherits these rules. Do NOT override unless the schema explicitly relaxes them.

**Frontmatter — mandatory fields (YAML):**

```yaml
---
id: <PREFIX><NNNN>        # per add-doc-ref-convention
type: <schema-type>       # one of the H3 types below
created: YYYY-MM-DD
updated: YYYY-MM-DD
related: []               # list of doc IDs this doc depends on or references
---
```

Optional: `by:` (last editor — human or agent identifier), `status:`, `slug:`.

**Body requirements:**

- **TL;DR** — H2 `## TL;DR` immediately after frontmatter, ≤60 words, extractive. Answers: what is this doc + why does it exist.
- **TOC** — required when the doc has >3 H2 sections. Flat bullet list linking anchors. Placed right after TL;DR.
- **Section chunks** — each H2/H3 body chunk ≈100-150 words. First sentence of every chunk is the topic sentence (grep target).
- **Refs** — doc→doc links MUST use `{{doc:<ID>}}` syntax (see `{{skill:add-doc-ref-convention/SKILL.md}}`). Command/skill refs use `{{cmd:}}` / `{{skill:}}` per `{{skill:add-resource-path-convention/SKILL.md}}`.
- **Metadata footer** — per `{{skill:add-documentation-style/SKILL.md}}` cache rules: update `updated:` on every write.

## Schema Registry

Fourteen canonical schemas. Each H3 below is the authoritative spec for that type. Commands MUST match these exactly. Each schema ≤150 words.

### feature-about

For `/add.new` (creates `docs/features/<slug>/about.md`).

- **Frontmatter:** `id: F[NNNN]`, `type: feature-about`, `slug:`, `status:`, `related: []`
- **Sections (ordered):** TL;DR · Problem · Users · Scope (Includes / Does NOT Include) · Success Metrics · References
- **Word caps:** TL;DR ≤60 · Problem ≤150 · Users ≤100 · Scope ≤200 · Metrics ≤100 · References unlimited (list only)
- **Compression:** Problem = topic sentence + extractive bullets. Users = table `role | goal | pain`. Scope = two bullet lists. Metrics = table `metric | target | source`. References = `{{doc:}}` / URL list.
- **Forbidden:** roadmap prose, aspirational language, inline design, code snippets >10 lines.

### feature-plan

For `/add.plan` (creates `plan.md`).

- **Frontmatter:** `id: F[NNNN]` (same as about), `type: feature-plan`, `related: [F[NNNN]]`
- **Sections:** TL;DR · Context (link `{{doc:F[NNNN]}}`) · Architecture Decisions · Tasks · Risks · Validation
- **Word caps:** TL;DR ≤60 · Context ≤80 · Decisions ≤200 (table) · Tasks unlimited (checklist) · Risks ≤150 (table) · Validation ≤100
- **Compression:** Decisions = table `decision | rationale | alternatives`. Tasks = `- [ ] area: action` bullets. Risks = table `risk | prob | mitigation`.
- **Forbidden:** duplicating about.md problem statement; narrative rationale outside the table.

### feature-design

For `/add.design` (creates `design.md`).

- **Frontmatter:** `id: F[NNNN]`, `type: feature-design`, `related: [F[NNNN]]`
- **Sections:** TL;DR · Screens · Components · Flows · Tokens · References
- **Word caps:** TL;DR ≤60 · Screens ≤200 (table) · Components ≤150 (list) · Flows ≤150 · Tokens ≤80 · References list only
- **Compression:** Screens = table `screen | purpose | primary action`. Components = bullets `name — shadcn source`. Flows = `step → step → step` sequences. Tokens = minified JSON.
- **Forbidden:** inline SVG, ASCII wireframes, >1 canonical example per component.

### hotfix-about

For `/add.hotfix` (creates `docs/hotfixes/<slug>/about.md`).

- **Frontmatter:** `id: H[NNNN]`, `type: hotfix-about`, `severity:`, `related: []`
- **Sections:** TL;DR · Symptom · Root Cause · Fix · Verification
- **Word caps:** TL;DR ≤60 · Symptom ≤100 · Root Cause ≤150 · Fix ≤150 · Verification ≤80
- **Compression:** Symptom = factual bullets `when / where / impact`. Root Cause = topic sentence + extractive bullets. Fix = bullets of file-level changes. Verification = checklist.
- **Forbidden:** blame narrative, long stack traces inline (link instead), post-mortem opinion.

### hotfix-related

For `/add.hotfix` (creates `related.md` listing impacted assets).

- **Frontmatter:** `id: H[NNNN]-related`, `type: hotfix-related`, `related: [H[NNNN]]`
- **Sections:** TL;DR · Impacted Files · Impacted Docs · Follow-ups
- **Word caps:** TL;DR ≤40 · all others = list only, no prose
- **Compression:** Files = bullets `path:line — reason`. Docs = `{{doc:}}` bullets. Follow-ups = `- [ ] action` checklist.
- **Forbidden:** duplicating about.md fix description; prose paragraphs.

### changelog

For `/add.done` (creates `docs/changelog/CHG[NNNN].md` or appends).

- **Frontmatter:** `id: CHG[NNNN]`, `type: changelog`, `date:`, `related: [F[NNNN] | H[NNNN]]`
- **Sections:** TL;DR · Changes · Breaking · Migration
- **Word caps:** TL;DR ≤50 · Changes ≤150 · Breaking ≤100 · Migration ≤100
- **Compression:** Changes = Conventional-Commit-style bullets `type(scope): summary`. Breaking = list or "none". Migration = steps.
- **Forbidden:** release marketing copy, subjective adjectives.

### brainstorm

For `/add.brainstorm` (creates `docs/brainstorm/YYYY-MM-DD-<slug>.md`). Date prefix is mandatory for chronological tree ordering.

- **Frontmatter:** `id: BRN-<slug>`, `type: brainstorm`, `related: []`
- **Sections:** TL;DR · Questions Explored · Candidate Directions · Open Threads
- **Word caps:** TL;DR ≤60 · Questions ≤150 · Directions ≤200 · Open Threads ≤100
- **Compression:** all sections = bullets only. Directions = `name — 1-line summary — pros/cons` bullets.
- **Forbidden:** committing to implementation, decisions (those belong in plan).

### owner

For `/add.init` (creates `docs/product/owner.md`).

- **Frontmatter:** `id: OWNER`, `type: owner`, `related: [PRODUCT]`
- **Sections:** TL;DR · Founder · Skills · Constraints · Goals
- **Word caps:** TL;DR ≤50 · each other ≤100
- **Compression:** Founder = `name / role / background` bullets. Skills = bullets. Constraints = table `constraint | impact`. Goals = bullets.
- **Forbidden:** narrative bio, personal opinions.

### product

For `/add.init` (creates `docs/product/product.md`).

- **Frontmatter:** `id: PRODUCT`, `type: product`, `related: [OWNER]`
- **Sections:** TL;DR · Vision · ICP · Core Value · Differentiators · Business Model
- **Word caps:** TL;DR ≤60 · Vision ≤80 · ICP ≤150 (table) · Core Value ≤100 · Differentiators ≤120 · Business Model ≤100
- **Compression:** ICP = table `segment | pain | JTBD`. Differentiators = bullets. Business Model = `{"pricing":..., "channels":...}` JSON.
- **Forbidden:** marketing fluff, superlatives, unverified claims.

### audit-report

For `/add.audit` (creates `docs/audit/<date>.md`).

- **Frontmatter:** `id: AUDIT-<date>`, `type: audit-report`, `related: [STACK]`
- **Sections:** TL;DR · Findings · Risks · Recommendations
- **Word caps:** TL;DR ≤60 · Findings ≤300 (table) · Risks ≤150 (table) · Recommendations ≤150
- **Compression:** Findings = table `area | severity | finding | evidence`. Risks = table `risk | prob | impact`. Recommendations = `- [ ] action — owner` checklist.
- **Forbidden:** narrative analysis, subjective ratings without evidence.

### diagnose-report

For `/add.diagnose` (creates `docs/diagnose/<slug>.md`).

- **Frontmatter:** `id: DIAG-<slug>`, `type: diagnose-report`, `related: []`
- **Sections:** TL;DR · Symptom · Hypotheses · Evidence · Recommended Route
- **Word caps:** TL;DR ≤60 · Symptom ≤80 · Hypotheses ≤150 · Evidence ≤200 · Route ≤60
- **Compression:** Hypotheses = table `hypothesis | likelihood | test`. Evidence = bullets `source → observation`. Route = single sentence + target command (`/add.hotfix` | `/add.new` | `extend` | `no-action`).
- **Forbidden:** speculation without evidence column, premature fix prescription.

### saas-copy

For `/add.copy` (creates `docs/copy/<slug>.md`).

- **Frontmatter:** `id: COPY-<slug>`, `type: saas-copy`, `related: [PRODUCT]`
- **Sections:** TL;DR · Headline · Subheadline · Value Props · CTA · Proof
- **Word caps:** TL;DR ≤40 · Headline ≤12 · Subheadline ≤25 · Value Props ≤120 · CTA ≤8 · Proof ≤100
- **Compression:** Value Props = bullets `benefit — mechanism`. Proof = bullets `metric / testimonial / logo`. All copy strings = quoted exact text.
- **Forbidden:** lorem ipsum, placeholder text, multiple variants (pick one canonical).

### landing-page

For `/add.landing` (creates `docs/landing/<slug>.md`).

- **Frontmatter:** `id: LAND-<slug>`, `type: landing-page`, `related: [COPY-<slug>, PRODUCT]`
- **Sections:** TL;DR · Sections · Components · Copy Refs · Assets
- **Word caps:** TL;DR ≤60 · Sections ≤150 (table) · Components ≤120 · Copy Refs list only · Assets list only
- **Compression:** Sections = table `order | section | purpose | component`. Components = bullets `name — shadcn source`. Copy Refs = `{{doc:COPY-<slug>}}` anchors. Assets = file paths.
- **Forbidden:** inline HTML, actual copy strings (reference saas-copy doc instead).

## Validation Gate Block

Every generator command MUST paste the following block as its final STEP. The block is self-contained and imperative. Replace `<SCHEMA>` with the schema type in use.

```markdown
## STEP N: Validation Gate (add-doc-schemas)

Run these checks against the doc you just wrote. DO NOT skip. DO NOT mark the command complete until every check passes or warns.

1. **Frontmatter presence.** Grep `^---$` at line 1. Confirm YAML block closes. Required fields for `<SCHEMA>`:
   - `id:` matches the prefix rule in {{skill:add-doc-ref-convention/SKILL.md}}
   - `type: <SCHEMA>` exact match
   - `created:` and `updated:` are ISO dates (YYYY-MM-DD)
   - `related:` is a YAML list (may be empty `[]`)
   If any field is missing: STOP. Fix the doc. Re-run this gate.

2. **TL;DR presence and cap.** Grep `^## TL;DR$`. Count words in the section body. MUST be ≤ cap declared in the schema H3 above. If over cap: rewrite extractively, do NOT summarize abstractively.

3. **TOC rule.** Count H2 sections (`^## `). If >3 and no `## TOC` / anchor list right after TL;DR: add it. Flat bullets only.

4. **Section word caps.** For each H2/H3 in the schema, count words in its chunk body. MUST be ≤ the schema's per-section cap. If over cap: split into sub-chunks or move detail to a linked doc. DO NOT compress via paraphrase.

5. **Doc refs resolve.** For every `{{doc:<ID>}}` in the doc, run reverse grep: `grep -rE "^id: <ID>$" docs/`. Each ref MUST return ≥1 hit. Unresolved refs = WARNING (not error); print them in the command output for the user to fix.

6. **Forbidden forms.** Confirm none of the schema's "Forbidden" items are present (emojis in headers, ASCII art, inline code >10 lines, aspirational language, abstractive paraphrase).

7. **Metadata footer.** Confirm `updated:` in frontmatter matches today's date. If editing an existing doc, confirm original `created:` was preserved.

DO NOT use abstractive summarization to fit caps. DO NOT delete content to pass the gate — split into linked docs instead. DO NOT silently drop unresolved refs — surface them as warnings.

Output at end:
- `gate: PASS` or `gate: FAIL`
- list of warnings (orphan refs)
- list of fixes applied (if any)
```

## Self-Governance

This skill is itself at risk of becoming unmaintainable. The following rules apply to `add-doc-schemas/SKILL.md`:

- **Each schema H3 ≤150 words.** If a schema needs more, split the type (e.g., `feature-plan-core` + `feature-plan-tasks`) rather than inflating one H3.
- **One type = one H3.** Do NOT nest sub-schemas inside a schema.
- **No inline examples of full docs.** Reference by `{{doc:}}` to a canonical example in `docs/examples/` (future).
- **Changes to this registry are breaking by default.** Bump the framework version and list affected commands.
- **When adding a new schema:** append to the registry in alphabetical-by-command order, register the type in the Validation Gate, and update `add-doc-ref-convention` only if a new ID prefix is introduced.
- **Audit trigger:** if the file exceeds 900 lines, run `/add.audit` on this skill and split.
