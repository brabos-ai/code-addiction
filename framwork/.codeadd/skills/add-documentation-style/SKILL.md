# Documentation Style Hub

## Overview

This skill is a **hub**, not a template library. It holds only the rules that apply to every ADD doc regardless of type: language policy, markdown formatting, cache discipline, metadata footer. Structural concerns (sections, word caps, frontmatter fields) live in `{{skill:add-doc-schemas/SKILL.md}}`. Reference resolution lives in `{{skill:add-doc-ref-convention/SKILL.md}}`. Token budget lives in `{{skill:add-token-efficiency/SKILL.md}}`.

The hub exists because universal rules were previously duplicated across per-type style files. Agents now load one small file for universal rules + one schema file JIT, instead of a monolithic style guide.

## When to Use

- Writing or editing any doc asset generated by an ADD command (about, plan, design, changelog, hotfix, owner, product, stack, audit, diagnose, copy, landing, brainstorm)
- Editing an existing generated doc (the cache rule below is mandatory)
- Reviewing a doc for universal style compliance before committing

## When NOT to Use

- PRDs authored by `/add.strategy` (framework meta-process, own template)
- External docs, vendor READMEs, third-party specs
- Commit messages, PR bodies — see `{{skill:add-commit/SKILL.md}}`
- Free-form brainstorm transcripts outside the `brainstorm` schema

## Universal Rules

These rules are schema-agnostic. They apply even when the schema is silent.

### Language

| Context | Language |
|---|---|
| Prose, explanations, rationale | PT-BR |
| Code, git, CLI flags, technical terms | English |
| Frontmatter keys and schema type names | English (machine-parsed) |

Technical terms (commit, branch, frontmatter, chunk, schema, hook) stay in English inside PT-BR prose. Do not translate them.

### Markdown Formatting

- **Headers:** no emojis, no decorative characters, no ASCII art. H1 is reserved for the doc title or schema type — never repeat.
- **Code fences:** always tagged with a language (` ```bash `, ` ```json `, ` ```yaml `). Untagged fences break agent parsing.
- **Tables:** use when you have ≥3 items with the same shape. Below 3, use bullets.
- **Inline code >10 lines:** forbidden — link to the file with `path:line` instead. Keeps the chunk under the 128-token budget.
- **Single line breaks** between sections. No triple blank lines, no horizontal rules as decoration.
- **No aspirational language** (*futuramente*, *idealmente*, *pretendemos*). Extractive docs describe what is, not what might be.

### Cache Documental (MANDATORY)

Every write to an existing doc MUST follow: **read → preserve → complement → bump `updated:`**. Never overwrite blind.

1. Read the existing doc in full before editing.
2. Identify sections already populated vs. template placeholders.
3. Preserve valid existing content; complement with new findings only.
4. On conflict: new information prevails, note the change in the relevant section.
5. Bump `updated:` in frontmatter to today's date on every write. `created:` is immutable.

| Situation | Action |
|---|---|
| Doc does not exist | Create from schema + set `created:` + `updated:` |
| Doc exists, template-empty | Fill sections + bump `updated:` |
| Doc exists, populated | Read → complement → bump `updated:` |
| Conflicting info | New info wins; log change in-section |

Anti-patterns: recreating a doc from scratch; silently overwriting populated sections; forgetting to bump `updated:`; duplicating content already owned by another doc (link instead).

### Metadata Footer

All generated docs carry the canonical frontmatter defined in `{{skill:add-doc-schemas/SKILL.md}}` (`id`, `type`, `created`, `updated`, `related`). Optional `by:` signals the editor (agent name or `human`). No separate trailing metadata block is required — the frontmatter IS the metadata.

## Delegation Map

When a concern is not covered above, load the relevant skill JIT. Do not reinvent.

| Concern | Load this skill |
|---|---|
| Schema, sections, word caps, validation gate | `{{skill:add-doc-schemas/SKILL.md}}` |
| Doc→doc refs, stable IDs, `{{doc:ID}}` syntax | `{{skill:add-doc-ref-convention/SKILL.md}}` |
| Token budget, per-zone compression, abstractive ban | `{{skill:add-token-efficiency/SKILL.md}}` |
| Command/skill/script refs (`{{cmd:}}` / `{{skill:}}`) | `{{skill:add-resource-path-convention/SKILL.md}}` |

## Workflow

1. Identify doc type → load matching schema from `add-doc-schemas`.
2. Load this hub for universal rules (already done if you are reading this).
3. If writing refs to other docs → load `add-doc-ref-convention`.
4. Apply compression per zone → `add-token-efficiency`.
5. Read existing file (cache rule) → write → bump `updated:` → run the schema validation gate.
